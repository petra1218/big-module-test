# 模型识别验证服务 - 设计文档

> 定位：这是一个用来验证"四川警察学院大模型服务"识别能力的内部工具型服务。目标是把一批图片送进模型、拿回识别结果、在原图上画框并并列展示，由人工判断模型靠不靠谱。遵循"够用就好、不要太复杂"的原则。

## 1. 运行环境与约定

- 操作系统：Ubuntu
- 语言/版本：Python 3.13.5
- 服务框架：FastAPI
- 监听地址：0.0.0.0:10086
- 启动方式：带启动脚本 start.sh，自动检测并安装运行库，以调试模式（--reload）启动
- 目录遍历：用 MinIO SDK（minio 客户端）按目录前缀列举文件，服务自行获取文件名列表（不需用户手列）
- 内部系统：凭证由前端页面填写，运行期驻留内存、不重复填写；不做前端到后端的密钥保密设计（仅限内网）
- 不引入数据库、不引入容器、不做登录鉴权

## 2. 接口契约要点（来自《对外接口-四川警察学院项目》）

### 2.1 认证
- POST /token/auth/login：body 带 appKey + appSecret，返回 data.accessToken、data.refreshToken
- POST /token/auth/refresh：body 带 refreshToken，返回新 accessToken
- POST /token/auth/logout：body 带 accessToken
- 拿到的 accessToken 用于 WebSocket 的 Sec-WebSocket-Protocol: Bearer xxxx（这是 token 的唯一用途）

### 2.2 发请求（Kafka 生产者，无需认证）
- Topic：Q_VEHICLE_FRONT_FACE_SHARE_TO_XQ（支持自定义）
- 消息字段：
  - ID：string，32 位，由我们生成，作为关联主键
  - DeviceID / IndexCode：二选一的必要参数（验证服务用一个固定测试设备值）
  - DeviceName：必填
  - SceneImageUrl：必填，图片地址（此处填 桶公开地址 + 目录 + 文件名 拼出的 MinIO 公开 URL）
  - CatTime：必填，格式 yyyy-MM-dd HH:mm:ss
  - Lat / Lng：可选
  - Expand：JSON 字符串，可放文件名等回显信息（兜底关联用）

### 2.3 收结果（WebSocket 消费者，Bearer 鉴权）
- 流水：ws://{ip}:{port}/apiWs/stream/data
- 预警：ws://{ip}:{port}/apiWs/alarm/data
- 两条连接都带 Sec-WebSocket-Protocol: Bearer xxxx
- 每条推送 payload 是 Base64 后的 JSON：先 base64 解码，再 JSON 解析
- 关键关联字段（两个通道都带）：kafkaReceiveImageId（回显我们发的 ID）、kafkaReceiveExpand（回显我们发的 Expand）

### 2.4 结果字段
- 流水含：themeLabel（主题标签）、ocrMessage（OCR 文本）、sourceName、inPic（输入图地址）、inTime/outTime/receptionTime（耗时计算）、isError、monitoringArea 等
- 预警含：alarmBoxs（检测框 JSON 字符串）、alarmLevel、themeLabel、inPic 等
- alarmBoxs 结构：[{"tag":"person","conf":0.948,"box":[x1,y1,x2,y2],"imgUrl":null}]
  - box 为绝对像素坐标（x1,y1 左上角，x2,y2 右下角），画框时按原图真实尺寸做缩放映射
  - tag / conf 为 null 时表示无标签/无置信度

## 3. 端到端数据流

1. 用户在 Web 页面填写配置并点击"开始验证"：MinIO 凭证（endpoint / access_key / secret_key / bucket / secure）、MinIO 公开基址、平台登录凭证（app_key / app_secret）、目录名。这些配置在**服务运行期间驻留内存**，后续操作无需重复填写；MinIO SDK 用凭证遍历该目录得到文件列表。
2. 后端登录拿 token；对每个遍历到的文件名：生成 32 位 ID，建立映射 ID 对应 文件名 对应 MinIO URL（= 桶公开地址 + 目录 + 文件名），组装接图消息（并发上限内）生产到接图 topic，状态记为"已发送"。
3. 两条 WS 长连持续接收：解码 base64 解析 JSON 取 kafkaReceiveImageId（兜底 kafkaReceiveExpand）回查映射拿到文件名与 MinIO URL 聚合（一张图可能对应多条流水/多条预警）。
4. 前端轮询结果；结果页左侧为图片列表（可点击切换），右侧详情区显示：该图原图 + canvas 画框图，并分两栏展示返回的流水信息（themeLabel / ocrMessage / sourceName / 耗时等）与预警信息（alarmBoxs 框、alarmLevel、themeLabel 等）。图片状态：等待中 / 已完成 / 有预警 / 超时。
5. 退出时调用 /logout。

## 4. 关联策略

- 由我们生成 32 位 ID（如 uuid4().hex，恰为 32 字符）写入请求 ID 字段，同时把文件名写入 Expand。
- 关联时优先用 kafkaReceiveImageId，为空则回退 kafkaReceiveExpand，双字段兜底确保能 join。
- 内存维护 ID 到 {filename, minio_url, status, stream[], alarms[]} 映射（轻量、无需持久化，进程存活期间保留，跨多次验证运行不丢失）。

## 5. 结果页布局与画框方案（前端）

- 整体布局：左「图片列表」+ 右「详情区」。
- 图片列表：展示本次所有图片（缩略图或文件名 + 状态徽标），点击切换到该图的详情。
- 详情区上半部：左 `<img>` 直接加载 MinIO 原图；右 `<canvas>` 加载同一原图，按 alarmBoxs 坐标画框标 tag/conf。
- 详情区下半部：分两栏——
  - 流水信息：themeLabel、ocrMessage、sourceName、inPic、inTime/outTime/receptionTime（耗时）、isError 等；
  - 预警信息：alarmBoxs（逐个框的 tag/conf/坐标）、alarmLevel、themeLabel 等。
- 画框：naturalWidth/Height 与显示尺寸求缩放比 sf，对 alarmBoxs 每个框绘制 rect(x1*sf, y1*sf, (x2-x1)*sf, (y2-y1)*sf) 并标注 tag + conf。alarmBoxs 是 JSON 字符串需二次 JSON.parse；imgUrl 为 null 忽略。
- 仅预警通道带框；若某图只在流水有记录、未命中预警，则右图无框（标"无检测框"）。

## 8. 配置项

- 配置来源：config.yaml 仅作可选默认值；运行时凭证、桶地址、目录名由**前端页面填写**，提交后存于服务内存（runtime_config），进程存活期间保留、不重复填写；前端另用 localStorage 记住填写内容，刷新页面不丢。
- 前端表单字段（运行时填写，驻留内存）：
  - 平台登录：app_key / app_secret
  - 平台地址：api_base（如 http://{ip}:{port}）、ws_base（如 ws://{ip}:{port}）
  - MinIO：endpoint / access_key / secret_key / bucket / secure（用于 SDK 遍历目录）
  - MinIO 公开基址：minio_public_base_url（如 http://minio-host:9000/bucket/，与"目录名 + 文件名"拼成 SceneImageUrl，模型可直连）
  - 目录名：directory（列举前缀）
- 服务内置（或 config.yaml 默认）：
  - host / port：默认 0.0.0.0 / 10086
  - kafka_bootstrap_servers：broker 地址（已确认无需 SASL）
  - topic_receive_image：Q_VEHICLE_FRONT_FACE_SHARE_TO_XQ（可自定义）
  - device_id / device_name：固定测试设备值
  - concurrency：并发发送上限（默认 5）
  - timeout_seconds：单图完成超时（默认 300）

## 9. 待向平台/实测确认的清单

1. 接图 topic 名为"车辆前脸"，是否真能接收任意图片做通用识别（还是只对车辆前脸分析）。
2. WS 推送 payload 究竟是 Base64(JSON) 还是纯 JSON（文档写 Base64，需实测首条）。
3. 每张发出的图是否保证至少产出一条流水（作为"完成"判据）；否则需超时兜底。
4. 字段名/嵌套是否与示例一致（建议先发 1 张图跑通端到端）。
5. alarmBoxs.box 坐标是否始终为原图绝对像素（示例值偏大，疑似原图尺寸，需确认是否随图缩放）。
6. token 有效期时长（决定 refresh 频率）与 WS 断线重连策略。
7. MinIO 遍历目录的凭证（endpoint/access_key/secret_key/bucket/secure）是否就绪；模型拉取 SceneImageUrl 的公开地址与列举用的内网地址是否一致。

## 10. 简化约束（不要过度设计）

- 不接数据库，映射与结果存内存即可（进程存活期间跨多次验证运行保留）。
- 后端不转发图片，浏览器直连 MinIO 公开地址。
- 不做自动准确率统计（无真值），仅可视化辅助人工判断。
- 不做用户系统、不做容器化、不做高可用。
- 内部系统：凭证由前端填写并驻留内存，不做前端到后端的密钥保密设计（仅限内网使用）；前端用 localStorage 记住填写内容。
- 错误处理以"标记状态 + 打日志"为主，不引入复杂重试框架。
- 目录遍历只做一层前缀列举（递归深度按需，默认列举该目录直接文件，如需子目录再扩展）。
